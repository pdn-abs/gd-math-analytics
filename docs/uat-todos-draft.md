User acceptance testing is usually done a week  before every realease.
To take a list of all the UAT listed tasks connected with the current release.
To ensure all the tasks related to the current release are UAT listed, there might be some tasks which are in TEST completed which are connected with the upcoming release, change those tasks status to UAT listed.
Among the UAT listed tasks some tasks which cannot tested by users manually or does not need UAT such task can be either Test archived or UAT archived based on the nature of the task.
Now from the remaining tasks we have to curate the list of levels (titles) which has to undergo UAT.
Based on based on the number of levels to be tested we can have one or more UAT session and each session duration can be 60 to 90 minutes long.
To conduct a UAT meeting the participants of the UAT should be informed a day prior to the meeting regarding the time plot of the UAT session.
Every UAT session should be recorded and the recorded meeting link is added along with every feedbacks in the UAT feedback sheet.
Before starting a UAT meeting make sure to have a UAT build ready for testing.
Once we start the UAT testers will be provided with the latest UAT build link with they need to install.
Then testers are provided with the level titles they need to check.
Split the levels that need to be tested between the testers and provide them there list of level titles.
Once the testers start playing the level and start sharing the feedbacks note down the feedback with the tester name or ID and time stamp it is conveyed. The best practice would be asking the testers to share their screen while giving a feedback.
Close the UAT sessions on time.
The collected feedback should be entered in the UAT feedback sheet.
Major bugs noticed should be informed immediately to the project manager.
Once all the UAT listed tasks are checked in UAT sessions then tasks which have successfully passed the UAT has to moved to Release- listed and depends on id  should be changed to the person whose is responsible for release of the app( in my case it was SRDJ).
Align with VC and KPK on collected feedbacks and change the status of feedbacks accordingly.
Add tasks for feedbacks which are in planned status(tasks which we have decided to plan for a fix).
